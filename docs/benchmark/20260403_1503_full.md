# Full Dataset Benchmark — 2026-04-03 (Post-Optimization)

Dataset: 100% http_logs (247M docs) | Warmup: 1 | Iterations: 10 | Mode: http_log_basic (no force-merge, no scroll)
Heap: 32GB | Single node

## Code Changes

Three optimizations applied to `ParquetDocValuesReader.java` and `ParquetDocValuesWriter.java`:

### 1. SORTED_SET Singleton Optimization (`ParquetDocValuesReader.java`)

**Problem**: For single-valued SORTED_SET fields (e.g. `clientip`), every document allocated a separate `long[1]` array to store its ordinal. At 247M docs, this created ~741M tiny arrays consuming ~13GB of heap (16 bytes object header + 8 bytes data per array).

**Fix**: Added `singletonOrds` field to `CachedSortedSet`. The constructor detects when all documents have exactly one ordinal and flattens `long[docCount][]` into a single `long[docCount]` array. This mirrors the existing `CachedSortedNumeric.singletonValues` pattern.

**Impact**: ~8GB heap savings per singleton SORTED_SET field (80% reduction). Eliminates Old Gen GC entirely.

### 2. Selective warmUp() (`ParquetDocValuesReader.java`)

**Problem**: `warmUp()` eagerly decoded ALL fields (NUMERIC, SORTED_NUMERIC, SORTED, SORTED_SET, BINARY) at segment open time. High-cardinality SORTED_SET fields like `request.raw` (~12M unique URLs) caused 82s Young Gen GC, 38.6s Old Gen GC, and 22.7% indexing error rate.

**Fix**: Only eagerly warm up fields with cheap decode paths:
- NUMERIC: uses mmap'd packed values (no heap)
- SORTED_NUMERIC: uses mmap'd packed values (no heap)
- SORTED: small dictionaries (geoip.country_name, geoip.city_name)

SORTED_SET and BINARY are decoded lazily on first query access.

**Impact**: Young GC 82s -> 33s, Old Gen GC 38.6s -> 0s, indexing errors 22.7% -> 0%.

### 3. V6 Flat Packed Format (`ParquetDocValuesWriter.java`, `ParquetDocValuesReader.java`)

**Problem**: The V5 block-packed format stored per-block DirectReader instances (15,000+ for 247M docs). Each value access required 3 array lookups + virtual dispatch: `blockMinValues[b] + blockReaders[b].get(offset) * blockGcds[b]`.

**Fix**: New VERSION_FLAT_PACKED (V6) format decouples value storage from skip index:
- **Block skip index**: `numBlocks x (long min, long max)` — 16 bytes/block, used by DocValuesSkipper
- **Flat packed section**: global `minValue`, `gcd`, `bitsPerValue` + single DirectWriter — one contiguous mmap region

Value access becomes: `globalMin + flatReader.get(idx) * globalGcd` (1 mmap read + 2 arithmetic ops, identical to Lucene90).

Backward compatible: V2-V5 indices remain readable via version-checked branches.

**Trade-off**: For timestamp fields spanning months, global bpv (~33) is larger than per-block bpv (~23), adding ~294MB to the packed section. This causes modest regression on full-scan sort operations but dramatically improves search_after queries that rely on DocValuesSkipper block pruning.

## Remaining Issues

1. **match-all +103%**: First query triggers lazy SORTED_SET decode for `_source` reconstruction. Needs background async warmUp or pre-fetch strategy.

2. **desc_sort_timestamp +53%**: V6 flat packed uses global bpv (33 bits) vs per-block (23 bits) for timestamps. DESC sort on monotonic data scans more values before BKD competitive pruning activates, amplifying the per-bit overhead.

3. **P90 cold-start spikes**: The lazy warmUp defers SORTED_SET decode to query time. The first query per segment pays ~100-500ms decode cost. Solutions: (a) async background warmUp thread, (b) increase benchmark warmup iterations, (c) SORTED_SET packed ordinals on the write path.

4. **asc_sort_size +39%**: Likely single-run noise (this was -7% in a prior run with the same code).
